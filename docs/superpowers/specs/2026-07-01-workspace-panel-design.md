# 工作区面板设计

## 背景

当前前端已经具备基础的 `worktree` 状态显示和预览链接能力，但这些信息分散在顶部状态位和后端接口中，缺少统一的可视化承载区。与此同时，`git` 相关信息尚未进入前端，用户无法在同一视图里理解“当前 worktree 是什么、改了哪些文件、每个文件改了什么、是否可以直接预览”。

本次设计不做“完整 Git 客户端”，而是围绕“查看改动”这个主任务组织信息结构。`worktree` 和 `git diff` 不拆成两个模块，而是合并成一个“工作区面板”，把 `worktree` 视作隔离工作区，把 `git` 视作该工作区里的改动状态。

## 目标

在现有聊天界面右侧增加一个统一的“工作区”视图，与现有 Agent 区域并列为同级标签，优先解决以下问题：

1. 用户能看到当前会话是否已经创建 `worktree`。
2. 用户能看到该 `worktree` 的分支、目录、`clean/dirty` 状态。
3. 用户能看到当前改动文件列表，而不是只看到一个抽象的 `dirty` 标签。
4. 用户能在前端直接查看 diff 预览。
5. 用户能在有预览地址时直接打开修改后的应用。

## 非目标

本次不包含以下内容：

1. 不做完整的 Git 提交工作流，不提供 `stage/unstage/commit/push` 的完整 UI。
2. 不做多 worktree 管理器，不展示跨会话 worktree 列表。
3. 不做三栏 IDE，不引入文件树浏览器或代码编辑器。
4. 不做复杂 diff 导航能力，例如“按 hunk 跳转”或“并排双栏 diff”。

## 推荐方案

采用“右侧统一工作区面板”，将当前右侧区域改造成双标签结构：

1. `参与 Agent`
2. `工作区`

默认仍停留在 `参与 Agent`，但当会话进入改代码模式、产生 `worktree` 或用户显式切换时，可以进入 `工作区` 视图。`工作区` 视图以改动预览和 diff 展示为中心，`git` 操作只作为低频辅助入口存在。

## 信息架构

`工作区` 面板自上而下分为四层。

### 1. 工作区摘要区

显示当前会话的 `worktree` 基础状态：

1. 分支名
2. `clean/dirty`
3. worktree 目录
4. 预览入口（如果存在 `previewUrl`）

这一层解决“当前是不是在隔离环境里工作”的问题，避免用户把会话状态和普通仓库状态混淆。

### 2. 改动概览区

展示本次工作区的变更摘要：

1. 改动文件总数
2. 未跟踪文件数
3. 是否存在文本 diff

如果当前 `worktree` 是 `clean`，这一区域直接显示空状态，而不是继续展示空白 diff 视图。

### 3. 文件列表区

以文件为粒度展示变更项，作为 diff 的导航入口。每个条目至少包含：

1. 文件路径
2. 文件状态标签：`modified` / `untracked`
3. 是否为当前选中项

文件列表不需要一开始做成真正的树结构。当前项目体量不大，扁平列表足够，而且更适合控制实现复杂度。

### 4. Diff 预览区

显示当前选中文件的 patch 预览。首版采用统一文本 diff 视图：

1. 文件头
2. `+` / `-` / 上下文行
3. 语义着色
4. 横向滚动

不做并排双栏，只做单栏 patch 预览。原因是右侧面板宽度有限，并排布局会明显压缩可读性。

## 交互设计

### 标签切换

右侧面板顶部增加二级标签：

1. `参与 Agent`
2. `工作区`

切到 `工作区` 后，保留当前会话上下文，不影响消息列表位置和输入区状态。

### 工作区加载

进入 `工作区` 标签时触发数据加载：

1. 先请求 `worktree status`
2. 再请求 `worktree diff`
3. 前端从 diff 文本中提取文件列表

如果没有 `worktree`，则展示引导态：

1. 当前会话尚未创建 worktree
2. 勾选“改代码”后发起任务即可进入隔离工作区

### 文件选择

当 diff 中包含多个文件时：

1. 文件列表默认选中第一个改动文件
2. 点击列表项切换右侧 diff 内容
3. 如果某个文件是未跟踪文件，显示其完整新增 diff

### 预览入口

如果 `previewUrl` 存在：

1. 在摘要区显示“打开预览”按钮
2. 样式高于“丢弃 worktree”，但低于 diff 本身

预览是重要入口，但不是当前面板的主任务，因此不占据面板主体区域。

### 丢弃 worktree

“丢弃 worktree”作为危险操作放在底部，不放在顶部主操作位。触发前应增加确认步骤，防止误删。

## 数据流

当前后端已具备以下接口：

1. `GET /api/sessions/:id/worktree/status`
2. `GET /api/sessions/:id/worktree/diff`
3. `POST /api/sessions/:id/worktree/discard`

首版前端不要求后端新增 diff 结构化接口。前端可以先消费现有纯文本 diff：

1. 通过 `status` 获取工作区元信息和 `previewUrl`
2. 通过 `diff` 获取完整 patch 文本
3. 在前端解析 `diff --git` / `+++` / `---` 片段，提取文件级块

这种方式的优点是改动小，能快速打通端到端体验。后续如果 diff 解析复杂度过高，再考虑后端补充结构化接口。

## 组件拆分

建议新增或重组以下前端单元：

1. 右侧面板标签状态：Agent / 工作区
2. `renderWorkspacePanel()`
3. `loadWorkspaceState()`
4. `parseUnifiedDiff()`
5. `renderWorkspaceFileList()`
6. `renderWorkspaceDiff()`

其中 `parseUnifiedDiff()` 只承担文件级拆分，不承担完整 diff AST 解析，控制复杂度。

## 空状态与异常处理

需要显式覆盖以下场景：

1. 无当前会话：显示“暂无工作区”
2. 当前会话未创建 worktree：显示引导态
3. worktree 已存在但 `clean`：显示“当前无改动”
4. diff 接口失败：显示错误态和刷新入口
5. diff 文本为空但状态为 dirty：显示“改动暂不可预览”，避免前端无反馈

## 视觉方向

沿用当前正在收敛的“控制台式协作界面”风格，不再引入 IDE 风格的复杂边界。

工作区面板应当满足：

1. 信息密度略高于聊天区，但仍属于同一设计系统
2. 文件列表和 diff 卡片复用现有卡片语言
3. `dirty`、新增、删除、危险操作使用统一语义色
4. diff 区域保留等宽字体，但面板标题与摘要区继续使用正文体系

## 测试策略

首轮实现至少覆盖：

1. 前端静态结构测试：工作区标签和容器存在
2. 前端行为测试：会请求 `status` 与 `diff`
3. diff 解析单元测试：多文件 patch 能拆成文件列表
4. 空状态测试：无 worktree / clean / diff 失败三类状态

## 风险与取舍

### 风险 1：纯文本 diff 前端解析容易脆弱

取舍：首版先接受这一复杂度，因为当前后端接口已经具备，能够最快形成可用结果。只要解析目标限制在“文件级拆分 + 原样展示 patch”，风险可控。

### 风险 2：右侧区域空间有限

取舍：避免三栏布局和双栏 diff，只做“文件列表 + 当前文件 patch”的紧凑视图。

### 风险 3：Agent 面板与工作区面板争夺右侧入口

取舍：使用标签切换，而不是同时并列展示两个完整区域。

## 结果定义

当实现完成后，用户应能在同一个右侧面板体系里回答以下问题：

1. 这次对话有没有 worktree？
2. 当前 worktree 是哪个分支？
3. 改了哪些文件？
4. 某个文件具体改了什么？
5. 能不能直接打开预览？

如果以上五个问题都能在前端直接回答，这个设计就达成目标。
