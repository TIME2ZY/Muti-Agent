# Agent-to-Agent 路由规则

你是多 Agent 协作系统中的一个 Agent。当你的任务完成或需要其他 Agent 介入时，通过 `@AgentName` 自动将任务路由给下一个 Agent。

## 当前 Agent 阵容

| Agent | 职责 | 何时 @ 它 |
|-------|------|-----------|
| **@Codex** | 任务编排、架构设计、合入前最终检查 | 需要方案设计、任务拆分、合入前确认 |
| **@小智** | 编程实现 | 需要写代码、修 bug、实现功能 |
| **@小虎鲸** | 测试、杂务 | 需要写测试、跑脚本、辅助任务 |
| **@M-M** | 代码 Review | 代码写完了需要 review、修复后需要确认 |

## 出口检查（发送前必须执行）

每条消息发送前，必须做一次"出口检查"：

```
回复前问自己："到我这里结束了吗？"
```

- **如果还需要下一个 Agent 采取行动** → 直接 @ 对方（跳过下面两问）
- **如果不需要别人行动** → 再问：
  1. **对方需要知道这个信息吗？**
  2. **这会影响对方的工作吗？**
  3. 两个都否 → 不 @

**关键设计：Q1 是短路的。** "需要对方采取行动"直接 @，不被 Q2/Q3 拦截。这防止"该 @ 不 @"的链条断裂。

## 格式要求（违反不触发路由）

```
@AgentName

[清晰的任务描述，包含完整上下文]
```

- **行首**写 `@AgentName`（前面不能有其他文字）
- 后面跟上明确的任务指令
- 附上对方需要的代码、日志、上下文

## 标准协作流程

### 开发新功能
```
用户: @Codex 实现用户登录功能
  → Codex: 拆解任务，设计方案
  → @小智 请实现以下登录功能：1. POST /api/login... 2. JWT token...
  → 小智: 写代码
  → @M-M 请 review 以上登录实现，关注安全性和错误处理
  → M-M: review 代码
  → @小智 发现 2 个 P1 问题：密码哈希未加盐、token 无过期时间。请修复
  → 小智: 修复
  → @M-M 已修复，请确认
  → M-M: 确认通过
  → @Codex review 通过，可以合入
```

### 修 Bug
```
用户: @小智 修复登录超时后不跳转的问题
  → 小智: 定位问题，修复代码
  → @小虎鲸 请为以下修复编写单元测试：...（修复的代码和边界条件）
  → 小虎鲸: 编写测试
  → @M-M 修复+测试已完成，请 review
  → M-M: LGTM
```

## 禁止行为

- ❌ 在代码块内部写 @mention（会被剥离，不触发路由）
- ❌ 在句子中间写 @mention（如"我建议@M-M 看看"→ 行首才触发）
- ❌ @ 自己（会被过滤）
- ❌ 无上下文地 @（如只写"@M-M"不加任何说明）
- ❌ 连续 @ 多个 Agent 做同一件事（选一个最合适的）
